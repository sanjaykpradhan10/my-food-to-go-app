### `Ticket` aggregate refactor (PR #11)

Full brainstorm → spec (`docs/superpowers/specs/2026-07-20-ticket-aggregate-design.md`) → plan (`docs/superpowers/plans/2026-07-20-ticket-aggregate.md`) → inline execution (6 tasks) → multi-agent code review → merge.

**Scope decisions made during brainstorming:**
- Add the full restaurant-worker REST lifecycle (`accept`/`preparing`/`readyForPickup`/`pickedUp`) even though nothing in the app previously drove those transitions — chosen over staying within the 3 pre-existing saga-driven states, since exercising the full book state machine was the point.
- No staff-initiated `reject()` — cancellation stays saga-driven only, no existing caller or policy for a manual override.
- Class-per-event domain events (`TicketCreatedEvent`, `TicketAcceptedEvent`, etc., behind a sealed `TicketDomainEvent` interface) over a flat event record — matches the book's actual Listing 5.1 pattern.
- A book-style `TicketDomainEventPublisher` (Listing 5.6–5.9 pattern) over inline pattern-matching in `TicketService`.

**What got built:** `Ticket` (`ftgo-kitchen-service`) is now a real DDD aggregate — a `TicketState` enum (`CREATE_PENDING → AWAITING_ACCEPTANCE → ACCEPTED → PREPARING → READY_FOR_PICKUP → PICKED_UP`, or `CANCELLED` legal only from the first three) replacing the old unguarded `String status`; state-changing methods (`confirm`, `accept`, `preparing`, `readyForPickup`, `pickedUp`, `cancel`) enforce legal transitions and return `List<TicketDomainEvent>`. `cancel()` extends the book's own two-tier exception distinction (`TicketCannotBeCancelledException` from `READY_FOR_PICKUP` vs. generic `UnsupportedStateTransitionException` elsewhere) to this app's extra pre-accept states. A new `TicketDomainEventPublisher` translates the typed domain events back into the pre-existing flat `KitchenEvent` wire record before writing to the outbox — a deliberate constraint, since `order-service`'s existing Kafka consumer deserializes `kitchen.events` with Jackson's default strict unknown-property handling and would break if any new field (e.g. `readyBy`) leaked onto the wire. `TicketService` shrank to a thin orchestrator (look up aggregate → call one method → save → publish). A new `TicketController` — kitchen-service's first REST controller — exposes the worker lifecycle, with `TicketNotFoundException`→404 and `UnsupportedStateTransitionException`→409.

**A real implementation-order constraint discovered mid-execution:** `Ticket.java` and `TicketService.java` are compilation-coupled (same Gradle module/source set) — rewriting `Ticket` alone left the whole module failing to compile until `TicketService` was updated too, so Tasks 2–4 (aggregate, publisher, service) had to be implemented together before any test could run, even though they were still committed as 3 separate commits matching the plan's task boundaries.

**Testing:** 43 new/updated tests — 19 aggregate state-machine tests (every legal transition + every illegal one), 3 publisher tests (including one asserting `readyBy` never leaks onto the wire), 13 updated service tests, 8 new `@WebMvcTest` controller tests. Full `./gradlew :ftgo-kitchen-service:build` green throughout.

**Docker e2e verification** (all in a fresh `docker compose up --build` stack): choreography happy path (`Order.APPROVED`, `Ticket.AWAITING_ACCEPTANCE`), the full new REST lifecycle driven manually to `PICKED_UP`, a 409 on an illegal transition and a 404 on an unknown ticket, wire-format byte-for-byte compatibility confirmed via direct outbox-row inspection, orchestration happy path (`SAGA_MODE=orchestration`, same end states via `SagaReply` translation), one compensation case (inactive consumer → `Order.REJECTED`/`Ticket.CANCELLED`), and a redelivery/idempotency spot-check (`processed_events` count and per-order ticket counts unchanged after a kitchen-service restart).

### Multi-agent code review (5 parallel Sonnet reviewers + Haiku confidence scoring)

Ran the full `code-review` skill pipeline against PR #11: CLAUDE.md-compliance audit, shallow bug scan, git-history review, prior-PR-comment check, and code-comment-compliance check, each as an independent agent. Found 3 candidate issues, each scored 0–100 by a separate Haiku confidence agent per the skill's rubric:
- Uncaught `UnsupportedStateTransitionException`/`TicketCannotBeCancelledException` from `Ticket.confirm()`/`cancel()` propagating out of the choreography Kafka listeners (`handleAccountingEvent`/`handleConsumerVerificationFailed`) with no try/catch — **75/100**, real but structurally prevented today by accounting-service's `SagaJoinService` gating (`CardAuthorized` can't arrive before a successful consumer verification, so the race that would trigger it isn't currently reachable).
- Missing WHY comment on `TicketDomainEventPublisher`'s deliberate `readyBy`-drop when mapping `TicketAcceptedEvent` to the wire format — **75/100**, a real gap against CLAUDE.md's explicit "why a Kafka topic carries a particular event shape" example, but the reasoning is documented in the PR description and `CONTEXT.md`.
- Missing comment on `cancel()`'s two-tier exception split — **50/100**, judged largely self-evident from the exception class names themselves.

All three fell below the skill's 80-point posting threshold, so the review comment reported "no issues found" with both 75-point items noted as non-blocking observations (not filed as formal findings).
