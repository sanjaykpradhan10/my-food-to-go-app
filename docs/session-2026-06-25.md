### 3. Ch. 3 structured walkthrough (not yet read ‚Äî use this when resuming)

**Part 1 ‚Äî Communication styles (the decision matrix)**

2√ó2 grid: one-to-one vs one-to-many √ó sync vs async. Key insight: synchronous communication creates temporal coupling. Microservices should prefer async.

**Part 2 ‚Äî Sync: REST + circuit breaker + service discovery**

- REST is simple but causes cascading failures
- Circuit breaker (Resilience4j) breaks the cascade after N failures
- Service discovery: client-side (Eureka registry) vs server-side (Kubernetes DNS ‚Äî what we'll use)

**Part 3 ‚Äî Async: Messaging + transactional outbox (most important)**

- Messages sent to channels (Kafka topics); types: commands / events / queries
- **The hard problem:** DB commit + Kafka publish are two operations ‚Äî can fail independently
- Three solutions:
  - **Transactional outbox** ‚Äî write event to `outbox` table in same DB tx, separate publisher forwards to Kafka
  - **Polling publisher** ‚Äî thread polls outbox and publishes
  - **Transaction log tailing** ‚Äî read MySQL binlog (CDC via Debezium) ‚Äî more efficient, more infra
- Eventuate Tram implements transactional outbox + polling publisher for us

**What we'll implement after reading:**
1. REST endpoint on `ftgo-order-service` (`POST /orders`) ‚Äî sync comms
2. Add Eventuate Tram + Kafka wiring to services
3. Publish `OrderCreated` event from `ftgo-order-service` ‚Äî demonstrates transactional outbox
