## What we did

### Ch.4 Create Order saga — choreography style (PR #6)

Full brainstorm → spec → plan → subagent-driven-development cycle, executed as 9 tasks by fresh implementer subagents with a task-scoped review after each and a final whole-branch review at the end.

**Scope decisions made during brainstorming:**
- Full 5-service saga (order, kitchen, consumer, accounting — delivery-service stayed out of scope, per the book's Create Order saga).
- Choreography now; orchestration deliberately deferred as a follow-up exercise to compare the two styles directly against the same use case.
- Full compensation matrix (all three failure points get a real compensating transaction), not just the happy path.
- One Kafka topic per service, each backed by its own outbox table — extends Ch.3's transactional-outbox pattern uniformly instead of special-casing new services.

**Built:**
- **consumer-service** and **accounting-service** get their first real code (previously empty Spring Boot stubs). consumer-service verifies a seeded consumer (id 1 active, id 2 inactive) and publishes `ConsumerVerified`/`ConsumerVerificationFailed`. accounting-service's `SagaJoinService` is the trickiest piece of the whole saga — it waits for both `ConsumerVerified` and `TicketCreated` (published by two independent services, arriving in either order on two separate Kafka consumer threads) via a local `SagaJoinState` table before authorizing a card, and resolves exactly once regardless of arrival order.
- **order-service**: `consumerId` added throughout (was previously absent from the schema entirely — a gap the spec had to fill in to let consumer-service have something to verify); `Order` now created in `APPROVAL_PENDING` instead of synchronously `APPROVED`; a new `OrderSagaService` plus three Kafka listeners transition it to `APPROVED` (only on kitchen's `TicketConfirmed`) or `REJECTED` (on any of three failure events).
- **kitchen-service**: gets its first outbox/Kafka-producer infrastructure (it previously only consumed); `Ticket` creation is now capacity-gated (rejects `TicketCreationFailed` above a quantity threshold); reacts to both `accounting.events` and `consumer.events` to confirm or compensate (cancel) the ticket.
- Since no pricing data flows through any saga event (order-service validates prices against restaurant-service but never persists or forwards them), **total line-item quantity substitutes for "order total"** as the threshold input throughout — kitchen's capacity limit (20) and accounting's authorization limit (10) are deliberately different so both failure paths are independently triggerable by order size alone.
- Docker Compose wiring (Dockerfiles + compose.yml) for both new services.

**A real bug found and fixed via manual end-to-end verification, not guessed:**
- kitchen-service, consumer-service, and accounting-service each got a brand-new `OutboxPublisher` (`@Scheduled` poller) in this pass, but **none of their main application classes had `@EnableScheduling`** — only order-service's did (pre-existing from Ch.3). Without it, Spring never invokes `@Scheduled` methods at all, so all three pollers silently never ran and the saga stalled forever with outbox rows stuck unsent. This slipped through 8 rounds of task-level code review because it's an integration-level gap unit tests don't naturally cover — exactly what manual e2e testing exists to catch.
- Fixed, and a `SchedulingEnabledTest` regression guard (asserts a `ScheduledAnnotationBeanPostProcessor` bean exists in the context) was added to all four services so this class of bug fails fast in the test suite instead of silently stalling the saga. Verified the guard actually catches the regression (temporarily removed `@EnableScheduling`, confirmed the new test fails, restored it).

**Manually verified via Docker (after the fix):**
1. Happy path — order reaches `APPROVED`, ticket reaches `AWAITING_ACCEPTANCE`, authorization `AUTHORIZED`.
2. Case A (consumer verification fails, consumerId 2) — order `REJECTED`, ticket compensated to `CANCELLED`, no authorization ever recorded.
3. Case B (kitchen capacity exceeded, quantity 25) — order `REJECTED`, no ticket ever created, no authorization.
4. Case C (card authorization declined, quantity 15) — order `REJECTED`, ticket compensated to `CANCELLED`, authorization `DECLINED`.
5. Redelivery/idempotency — forced a Kafka redelivery (reset `sent_at` on an already-sent outbox row), confirmed the authorization count stayed at exactly 1 (no double-processing), proving both the `processed_events` dedup ledger and accounting-service's join-state resolve-once guard work correctly.

### Process note: infrastructure outage mid-session

Partway through Task 8, the platform's Bash/Agent safety classifier had an extended outage (roughly 30–40 minutes), blocking subagent dispatch and any mutating/networked Bash calls. Handled by falling back to direct execution (the controller ran verification commands and committed Task 8's already-correct files directly, then wrote the task report itself) rather than blocking indefinitely, and resumed normal subagent dispatch once the outage cleared. Separately, the host machine appears to have slept for about an hour during this window, which froze the Docker containers' clocks and Kafka consumer sessions — resolved by restarting the affected containers rather than debugging state from a suspended VM.

## What's implemented now (full picture)

- **ftgo-restaurant-service** (port 8085): unchanged from Ch.3.
- **ftgo-consumer-service** (port 8081): `Consumer` entity + seed data (id 1 active, id 2 inactive); consumes `order.events`, verifies, publishes `ConsumerVerified`/`ConsumerVerificationFailed` to `consumer.events` via its own outbox.
- **ftgo-order-service** (port 8082): creates `Order` in `APPROVAL_PENDING` (now with `consumerId`); `OrderSagaService` + three listeners transition it to `APPROVED`/`REJECTED` based on saga events.
- **ftgo-kitchen-service** (port 8083): creates `Ticket` in `CREATE_PENDING` (capacity-gated, limit 20 total quantity) on `OrderCreated`; confirms (`AWAITING_ACCEPTANCE`) or compensates (`CANCELLED`) based on `accounting.events`/`consumer.events`.
- **ftgo-accounting-service** (port 8084): `SagaJoinService` joins on `ConsumerVerified` + `TicketCreated` (either order, via `SagaJoinState` with `@Version` optimistic locking), authorizes (limit 10 total quantity), publishes `CardAuthorized`/`CardAuthorizationFailed` to `accounting.events`.
- **ftgo-service-registry** (port 8761): unchanged from Ch.3.
- **docker-compose**: 10 services total (mysql, zookeeper, kafka, kafka-connect, connector-registrar, service-registry, restaurant-service, order-service, kitchen-service, consumer-service, accounting-service).
