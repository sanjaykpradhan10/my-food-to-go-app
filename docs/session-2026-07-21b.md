## What we did

### Scope: Cancel Order saga, sub-project 2 of 3

The `Order` DDD aggregate (PR #12) left `Order.CANCEL_PENDING` — reachable via `POST /orders/{id}/cancel` — with nothing to resolve it. This session built the real cross-service saga that does: kitchen cancels the ticket, and only if kitchen confirms that's possible does accounting reverse the payment authorization. Both saga modes (choreography and orchestration), matching this project's established dual-mode-switchable pattern.

**Key design decisions from brainstorming:**
1. **Sequential, kitchen-gates-accounting, not parallel.** `Order.cancel()` is only legal from `APPROVED`, meaning the `Ticket` has already been confirmed and may be anywhere from `AWAITING_ACCEPTANCE` through `PICKED_UP` — `Ticket.cancel()` may legitimately reject (already `READY_FOR_PICKUP`, etc.). Accounting is only ever contacted if kitchen confirms cancellable; if kitchen rejects, `Order.undoCancel()` fires immediately and accounting is never touched — no compensating re-authorization step is ever needed.
2. **`Authorization` (accounting-service) became a DDD aggregate too**, going beyond Ch.5's two book-named worked examples (`Ticket`, `Order`), so "reverse" is a real guarded transition (`AUTHORIZED → REVERSED`) rather than an unguarded status string.
3. **No new join-instance table for orchestration mode.** Unlike Create Order saga (parallel join via `CreateOrderSagaInstance`), Cancel Order is a strict linear pipeline — `CancelOrderSagaOrchestrator` is stateless, driven purely by incoming replies, using `Order`'s own status as the implicit saga state.
4. **`SagaReply` gained a `sagaType` field** so order-service's one shared `saga.replies` listener can route between `CreateOrderSagaOrchestrator` and the new orchestrator, now that two saga types share the same `orderId` keyspace.

### Two real gaps found and fixed *during planning*, before any code was written

- **`KitchenCommand` needed its own `sagaType` field, not just `SagaReply`.** `TicketService.handleCancelTicketCommand`'s `"CancelTicket"` command turned out to be shared by both sagas — Create Order's existing compensation path (`CreateOrderSagaOrchestrator.sendCancelTicket`) and Cancel Order's new primary flow — so the handler can't statically know which saga a given command belongs to; it has to be told. Discovered while gathering context for the implementation plan, before any task was dispatched.
- **`accounting.commands` needed the same `KitchenCommand`-style generalization.** `AuthorizeCardCommand` had no `commandType` discriminator at all — safe only because that topic had only ever carried one command type. Adding `ReverseAuthorization` required renaming it to `AccountingCommand` with a real discriminator, mirroring `KitchenCommand`'s existing shape.

Both corrections were made to the committed spec doc before the plan was written, keeping the plan itself accurate from the start.

### A required fix bundled into kitchen-service (not new capability)

`TicketService.handleCancelTicketCommand` previously did `ticket.cancel(); ticketRepository.save(ticket);` with **no exception handling and no reply published** — a gap already flagged (non-blocking) during the `Ticket` aggregate's own PR review, since its only caller before this session (Create Order saga's compensation path) never inspected the outcome. This session's orchestration mode needs a real reply to know whether cancellation succeeded, so the fix (try/catch, reply `"TicketCancelled"`/`"TicketCancellationRejected"`) was now required, not optional.

### A third, more serious bug — caught only by hands-on Docker testing, not by review

`AuthorizationCancelService.reverse()` was built as one method shared by both the choreography listener (kitchen's `TicketCancelled` domain event) and the orchestration command listener (`ReverseAuthorization` command) — but it always published the reversal as a domain event on `accounting.events`. That channel has no consumer in orchestration mode (order-service's `AccountingEventListener` is choreography-only), so `CancelOrderSagaOrchestrator` was left waiting forever on a `saga.replies` message that never arrived. Every orchestration-mode cancellation got stuck permanently in `CANCEL_PENDING`.

This slipped through all 7 tasks' task-level reviews because `AuthorizationCancelServiceTest` mocked `AuthorizationDomainEventPublisher` and only verified it was called — the test never round-tripped a message between two actual services, so the "nothing consumes this topic in orchestration mode" gap was invisible to it. It was caught only during manual Docker e2e verification, when an orchestration-mode cancel request visibly hung in `CANCEL_PENDING` instead of settling to `CANCELLED`.

**Fix**: split into two methods, mirroring the choreography/command pattern every other saga participant in this codebase already uses (e.g. `TicketService.handleOrderCancelled` vs. `handleCancelTicketCommand`, `OrderSagaService`'s choreography path vs. `CreateOrderSagaOrchestrator`'s command path):
- `reverseForChoreography(eventId, orderId)` — unchanged behavior, publishes the domain event to `accounting.events`.
- `reverseForCommand(eventId, orderId, sagaType)` — new, publishes a `SagaReply` to `saga.replies` so the orchestrator can proceed.

Fixed post-hoc (after all 7 tasks were reviewed clean), with new tests for both methods, before the PR was opened.

### Docker e2e verification

Full multi-module build green throughout, including after the post-hoc fix. Verified via `docker compose`, both saga modes:
- **Choreography**: cancel success (order still `AWAITING_ACCEPTANCE`-stage ticket) → `Order.CANCELLED`, `Ticket.CANCELLED`, `Authorization.REVERSED`. Cancel rejection (ticket driven to `READY_FOR_PICKUP` via the existing restaurant-worker REST API first) → `Order` back to `APPROVED`, ticket unchanged at `READY_FOR_PICKUP`, `Authorization` confirmed still `AUTHORIZED` (accounting never contacted).
- **Orchestration**: same two scenarios, same end states — the second run (after the `AuthorizationCancelService` fix) is what actually confirmed the fix worked, since the first orchestration-mode attempt was the run that surfaced the bug.

## What's implemented now (full picture)

- **ftgo-order-service**: `OrderCancelSagaService` (choreography) and stateless `CancelOrderSagaOrchestrator` (orchestration) resolve `Order.CANCEL_PENDING`. `OrderController.cancel()` routed through a new saga-mode-aware `OrderCancellationSagaTrigger`.
- **ftgo-kitchen-service**: `TicketService.handleCancelTicketCommand` fixed to reply with the real outcome; new `handleOrderCancelled` for the choreography path; both guard `Ticket.cancel()`'s two exception types and publish/reply accordingly.
- **ftgo-accounting-service**: `Authorization` is now a DDD aggregate (Ch.5) — `AuthorizationStatus`, guarded `authorize`/`decline`/`reverse`, `AuthorizationDomainEventPublisher`. New `AuthorizationCancelService` with the split choreography/command methods. `AccountingCommandListener` (renamed from `AuthorizeCardCommandListener`) dispatches both `AuthorizeCard` and `ReverseAuthorization`.
- **ftgo-consumer-service**: unchanged this session beyond the `sagaType` wire-format scaffolding (its `publishReply` always tags `"CreateOrder"`, since it never participates in Cancel Order).
- **Everything else** (`Ticket`'s aggregate itself, `Order`'s aggregate itself, restaurant-service, service-registry, ftgo-common): unchanged this session.
