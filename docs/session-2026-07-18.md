### Shared `ftgo-common` module extraction (PR #8)

Closed the "optional, not blocking" cleanup item noted at the end of the prior session: the 4x-duplicated outbox/dedup infrastructure (`OutboxEvent`, `ProcessedEvent`, their repositories, `OutboxPublisher`, `KafkaProducerConfig`) copy-pasted across order/kitchen/consumer/accounting-service.

**Process:** full brainstorm ‚Üí spec (`docs/superpowers/specs/2026-07-18-shared-outbox-module-design.md`) ‚Üí plan (`docs/superpowers/plans/2026-07-18-shared-outbox-module.md`) ‚Üí subagent-driven-development (9 tasks, each independently reviewed) ‚Üí whole-branch review ‚Üí re-review after fixes.

**What got built:** a new Gradle module `ftgo-common` (package `com.sanjay.ftgo.common.outbox`), built as a plain library (`bootJar` disabled, `java-library`'s `api` config exposing JPA/Kafka types transitively). All four saga services depend on it via `implementation project(':ftgo-common')`. Saga wire-format records (`SagaReply`, `KitchenCommand`, etc.) deliberately stayed per-service ‚Äî infra-only extraction.

**Two real bugs found via Docker e2e verification, not unit tests:**
1. Putting `@EntityScan`/`@EnableJpaRepositories` directly on `FtgoOrderServiceApplication` broke `OrderControllerTest` (`@WebMvcTest` slice filtering doesn't exclude annotations on the primary configuration class). Fixed with a separate `PersistenceConfig` class per service, applied consistently across all four from the start once found in Task 5.
2. `@EntityScan`/`@EnableJpaRepositories` alone don't register `@Component`/`@Configuration` beans ‚Äî the shared `OutboxPublisher` and `KafkaProducerConfig` were silently never picked up by any service, no startup error, orders stuck in `APPROVAL_PENDING` forever. Fixed with `@ComponentScan(basePackages = "com.sanjay.ftgo.common.outbox")` on all four `PersistenceConfig` classes. Re-verified via Docker after the fix.

**Independent post-merge code review (8-angle, medium effort) found more, all fixed before considering the work done:**
- The hand-copied `@ComponentScan` itself was a maintainability risk (a future 5th consumer could omit it and reproduce the exact stuck-order bug) ‚Äî replaced with a real Spring Boot auto-configuration shipped from `ftgo-common` itself (`OutboxAutoConfiguration` via `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`), so any consumer gets the beans automatically. Re-verified via Docker again since this is exactly the class of change unit tests can't catch.
- Redundant dependency declarations (`spring-boot-starter-data-jpa`/`spring-kafka` declared twice across build files) ‚Äî cleaned up.
- `ftgo-common` was inheriting `spring-boot-starter-web` unnecessarily via the root `subprojects{}` block ‚Äî excluded, given its own minimal dependency set.
- `OutboxEvent.orderId` renamed to `aggregateId` (Java-level only, DB column unchanged) since the entity is now generic/shared, not order-service-specific.

### Spring Boot 3.5.3 ‚Üí 3.5.16 upgrade (PR #9 + PR #10)

Researched the latest Spring Boot 3.5.x release before touching anything: **3.5.16 is the final release of the 3.5.x line ‚Äî it reached open-source end-of-life on 2026-06-30**, flagged to the user before proceeding (explicitly chose to stay on 3.5.x rather than jump to the 4.x major line, which is a separate future decision).

Checked every documented 3.5.4‚Äì3.5.16 behavioral change (TaskExecutor bean renaming, heapdump endpoint default, stricter `.enabled` boolean parsing, profile-name restrictions, Redis/Prometheus/Groovy property renames, docker-compose SSL fixes) against this codebase ‚Äî none apply (no actuator, no `taskExecutor` bean, no `.enabled` properties, no `@Profile` usage, no Spring Boot docker-compose integration). Confirmed with three additional checks after a user follow-up question: zero new compiler deprecation warnings, no major-version transitive-dependency boundary crossed (Spring Framework/Hibernate/Jackson/Tomcat all stayed within their existing major lines), and zero `deprecated`-logged warnings in any of the 6 services' Docker startup logs.

Spring Cloud BOM bumped to `2025.0.3` (latest patch compatible with Boot 3.5.x ‚Äî `2025.1.x` requires Boot 4.0).

**PR #9 merged with an incomplete fix** ‚Äî the Spring Cloud BOM was only bumped in `ftgo-order-service/build.gradle`, missing that `ftgo-restaurant-service` and `ftgo-service-registry` (the actual Eureka client/server pair) declare the same BOM independently. This was caught by an independent 8-angle code review *while the review was still running* ‚Äî the PR merged on its own before the review finished (not a deliberate action), so the fix had to land as a second PR (#10) rather than a pre-merge amendment. Re-verified via Docker (Eureka registration/discovery specifically) after aligning all three modules on `2025.0.3`.
