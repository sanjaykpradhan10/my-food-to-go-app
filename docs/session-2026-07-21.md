## What we did

### Scope decision: the full Fig. 5.14 state machine, decomposed into 3 sub-projects

`Order` (order-service) was the book's other Ch.5 worked example, already flagged as remaining work after `Ticket`'s refactor. During brainstorming, two scope questions came up:

1. **Should the refactor cover only the existing create-order lifecycle (`APPROVAL_PENDING → APPROVED/REJECTED`), or the full book state machine including cancel/revise?** Chose the full Fig. 5.14 scope — cancel/revise are the book's other worked transitions and this project has consistently chosen to exercise the full model (mirroring the `Ticket` refactor's decision to add the restaurant-worker REST lifecycle).
2. **Given cancel/revise, how far should this session go — aggregate methods only, or full cross-service saga wiring (kitchen ticket adjustment, accounting refund/re-authorization)?** Full saga wiring is genuinely comparable in size to the original Ch.4 Create Order saga (done twice, once per style) — too large for one session. Decomposed into 3 sequenced sub-projects: (1) the `Order` aggregate itself, including REST endpoints that trigger cancel/revise even though nothing yet resolves the resulting pending states; (2) a **Cancel Order saga** (future session); (3) a **Revise Order saga** (future session, the book's trickiest saga). This session implemented sub-project 1 only.

### `Order` aggregate refactor (sub-project 1 of 3)

Full brainstorm → spec (`docs/superpowers/specs/2026-07-21-order-aggregate-design.md`) → plan (`docs/superpowers/plans/2026-07-21-order-aggregate.md`, 7 tasks) → subagent-driven-development → task-level review → Docker e2e verification.

**What got built:** `OrderStatus` gained 3 states (`CANCEL_PENDING`, `CANCELLED`, `REVISION_PENDING`) alongside the existing 3. `Order` gained 8 guarded state-changing methods — `noteApproved`/`noteRejected` (the existing create-lifecycle, now guarded and event-returning instead of unguarded setters), `cancel`/`noteCancelled`/`undoCancel` (the `APPROVED ⇄ CANCEL_PENDING ⇄ CANCELLED` branch), `revise`/`confirmRevision`/`rejectRevision` (the `APPROVED ⇄ REVISION_PENDING` branch) — each returning `List<OrderDomainEvent>` from a new sealed `OrderDomainEvent` interface with 8 class-per-event implementations. A new `OrderDomainEventPublisher` replaced the single-purpose `OrderCreatedEvent` wire record with a generic `OrderEvent` record (matching kitchen-service's `KitchenEvent` shape — one flat record, `eventType` discriminator, nullable fields depending on type) so `order.events` can carry all 9 event types on one topic, while keeping `OrderCreated`'s JSON byte-for-byte unchanged for existing consumers. Both saga participants — `OrderSagaService` (choreography) and `CreateOrderSagaOrchestrator` (orchestration) — were rewired to call the aggregate's guarded methods (catching `UnsupportedStateTransitionException` to preserve the old silent-ignore-on-already-resolved behavior) instead of duplicating an inline status guard. `OrderController` gained `POST /orders/{id}/cancel` and `POST /orders/{id}/revise`, calling the aggregate directly — a **known, deliberate temporary gap**: no saga yet resolves the resulting `CANCEL_PENDING`/`REVISION_PENDING` states, so calling these endpoints today strands an order until sub-projects 2/3 land.

**A required cross-service fix, caught during design:** `ftgo-kitchen-service`'s `OrderEventListener` deserialized every `order.events` message unconditionally as `OrderCreatedEvent` and called `handleOrderCreated`, with no `eventType` check — a latent bug that was harmless only because `order.events` had never carried anything but `OrderCreated` before. Once this session's other 8 event types start flowing, that listener would misinterpret e.g. an `OrderApproved` event as a new order with null `consumerId`/`restaurantId`/`lineItems` and create a bogus `Ticket`. Fixed with the same discriminator-first pattern already used by every other listener in this codebase (`KitchenEventListener`, `AccountingEventListener`, etc.) — a one-line `if (!"OrderCreated".equals(event.eventType())) return;` guard, with a WHY comment.

### Subagent-driven-development: two real coordination issues, both caught and resolved

Running the 7-task plan surfaced two genuine cross-task problems, both diagnosed and repaired by the controller rather than left for the implementer to improvise around:

1. **A real plan-sequencing gap.** Task 3 (which introduced `OrderDomainEventPublisher` and deleted the old `OrderCreatedEvent` wire record) hit a compile break in `ChoreographyOrderCreationSagaTrigger.java` — a file the plan had scoped to Task 5, since it was the only remaining consumer of the deleted type. Resolved by pulling Task 5's already-written rewrite of that one file forward into Task 3 (verbatim), so the module kept compiling; Task 5 then verified the pulled-forward code matched exactly and only did its own remaining work (`CreateOrderSagaOrchestrator`).
2. **A branch-history corruption from an implementer's first attempt.** Task 4's implementer (rewriting `OrderSagaService`) reported DONE, but its commit's parent turned out to be *before* Tasks 1–3 rather than after Task 3 — silently reintroducing the already-deleted `OrderCreatedEvent.java` as an unused file and orphaning three prior commits from the branch's reachable history (though they remained intact as dangling commit objects). Caught by the task reviewer flagging the unexplained reappeared file. Diagnosed by diffing the corrupted commit's tree against the last-known-good commit, confirmed the only real difference was the reintroduced file plus the two legitimately-changed files, and repaired by resetting the branch to the correct parent and reapplying just the two legitimate files' content — no implementer work was lost or redone.

All 7 tasks passed task-level spec-compliance and code-quality review clean (after these two fixes).

### Docker e2e verification

Full multi-module `./gradlew clean build` green, then verified via `docker compose`:
- **Choreography mode**: create-order happy path (`Order.APPROVED`); `/cancel` reaching `CANCEL_PENDING` with kitchen-service confirmed *not* reacting (ticket count for that order held at 1 across 12s of polling, directly validating the `OrderEventListener` fix); `/revise` reaching `REVISION_PENDING` with line items left unchanged (as designed — only `confirmRevision`, not yet built, actually applies them); 404 on an unknown order id; 409 on cancelling a non-`APPROVED` order; one compensation case (inactive consumer → `Order.REJECTED`/`Ticket.CANCELLED`); redelivery/idempotency (forced a kitchen-service restart, `processed_events` count and per-order ticket counts unchanged).
- **Orchestration mode** (`SAGA_MODE=orchestration`): happy path and the same compensation case, both reaching identical end states to choreography.
