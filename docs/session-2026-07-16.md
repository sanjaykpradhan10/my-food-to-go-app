### 2. Ch.3 async messaging + transactional outbox (PR #3)

Full brainstorm → spec → plan → subagent-driven-development cycle. Key decisions: hand-rolled outbox (not Eventuate Tram), `Order` persisted via JPA for the first time, kitchen-service (previously an empty stub) got its first real code.

**Built:** `Order`/`OrderLineItem` as JPA entities; `OutboxEvent` + `OutboxEventRepository` in the same package as `Order` (deliberate deviation from restaurant-service's infra-package convention, since `OrderService` needed direct persistence access for the transactional outbox guarantee); `OrderService.createOrder()` now writes `Order` + an `OutboxEvent` row in one `@Transactional` method; `OutboxPublisher` — a `@Scheduled` poller publishing unsent rows to Kafka topic `order.events`, keyed by `orderId` (a Kafka-key bug — keying by `eventId` instead of `orderId` — was caught in final review and fixed); kitchen-service's `TicketService`/`OrderEventListener`, deduping by `event_id` via a `processed_events` ledger (protects against Kafka's at-least-once redelivery); docker-compose fixed Kafka's listener config (previously only advertised `localhost:9092`, which silently breaks inter-container traffic — added a proper internal/external dual-listener setup).

Verified end-to-end via Docker: order placed → outbox row unsent → poller flips to sent → Ticket created → forced redelivery correctly deduped.

### 3. Ch.3 client-side service discovery (PR #4)

New `ftgo-service-registry` module (standalone Eureka server, port 8761). `restaurant-service` registers itself on startup. `order-service` resolves it dynamically via a `@LoadBalanced RestClient` (`http://ftgo-restaurant-service`, a logical name resolved by Spring Cloud LoadBalancer) instead of a hardcoded URL — **`RestaurantServiceProxy` itself was never touched**, only how its `RestClient` bean is built (`RestClientConfig.java`). Tests swap Eureka for Spring Cloud LoadBalancer's static `SimpleDiscoveryClient`, so `RestaurantServiceProxyTest` needed zero code changes either.

Verified end-to-end via Docker including instance eviction (`docker compose stop restaurant-service` → registry entry removed → circuit breaker degrades to 503) and dynamic recovery (restart → re-registers → order-service resumes 201s) **without restarting order-service**, proving discovery re-resolves per call rather than caching at startup.

### 4. Ch.3 transaction log tailing / CDC (PR #5) — closes the last Ch.3 IPC pattern

Debezium + Kafka Connect added as a second outbox delivery mechanism, switchable against the existing polling publisher via one `OUTBOX_PUBLISH_MODE` env var (default `polling`), reconciled independently by two idempotent points on every `docker compose up`: `OutboxPublisher`'s `@ConditionalOnProperty(outbox.publish-mode=polling, matchIfMissing=true)`, and a new one-shot `connector-registrar` container that PUTs/DELETEs the Debezium connector via Kafka Connect's REST API. The connector uses Debezium's Outbox Event Router SMT to republish the existing `outbox_events.payload` column unchanged to `order.events`, keyed by `order_id` — byte-identical to the poller's output, so **kitchen-service needed zero changes**.

MySQL binlog enabled (ROW format) + a dedicated least-privilege `debezium` replication user.

**Two real bugs found and fixed via hands-on verification, not guessed:**
- Plan's specified `debezium/connect:2.7` image tag doesn't exist → used `2.7.3.Final`.
- Plan's specified Kafka Connect host port `8083:8083` collided with kitchen-service's pre-existing mapping (a bug in the plan itself) → remapped host-side only to `8087:8083`.
- **The big one**: `debezium/connect:2.7.3.Final` reports connector status `RUNNING` via the REST API while **silently delivering zero events** — it issues the legacy `SHOW MASTER STATUS` SQL statement, which MySQL 8.4 removed (renamed to `SHOW BINARY LOG STATUS`). Fixed by bumping to `debezium/connect:3.0.0.Final`, verified via live logs showing correct binlog streaming. Both non-obvious pins (image version, port remap) now have inline `#` comments in `compose.yml` per the final review's recommendation, so a future "cleanup" doesn't reintroduce either bug.

Verified end-to-end: polling-mode regression check (unaffected by CDC infra being present), CDC-mode delivery (poller inert, `sent_at` stays NULL, Debezium alone delivers), and mode-switch replay safety (`processed_events` dedup ledger as the safety net — confirmed via logs that `snapshot.mode: no_data` meant no replay occurred at all).

## What's implemented now (full picture)

- **ftgo-order-service** (port 8082): `Order` persisted via JPA; `POST /orders` validates against restaurant-service (discovered via Eureka + circuit breaker); writes `Order` + `OutboxEvent` in one transaction; `OutboxPublisher` (polling) OR Debezium (CDC) delivers `OrderCreated` to Kafka topic `order.events`, switchable via `OUTBOX_PUBLISH_MODE`

- **connector-registrar**: one-shot container reconciling the Debezium connector's registration state to `OUTBOX_PUBLISH_MODE`
- **docker-compose**: 9 services total (mysql, zookeeper, kafka, service-registry, kafka-connect, connector-registrar, restaurant-service, order-service, kitchen-service)
